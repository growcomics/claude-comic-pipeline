# Project Folder Convention

The `comic-status-board` skill reads project state from a standard on-disk layout. This document is the canonical spec.

The convention is also written to by `reference-gathering`, `comic-production` (via `shotlist-driven-flow.md`), `continuity-check`, and `page-composer`. Every skill in the pipeline agrees on this layout.

---

## Project root layout

```
<project>/
├── shotlist.json                          # canonical plan from script-breakdown
├── shotlist.md                            # human-readable companion
├── STATUS.md                              # auto-generated by comic-status-board
├── STATUS-references-board.png            # composite, end of references stage
├── STATUS-generation-board.png            # composite, end of generation stage
├── STATUS-composition-board.png           # composite, end of composition stage
├── references/                            # populated by reference-gathering
│   ├── characters/<slug>/
│   │   ├── face-card.png                  # canonical face ref
│   │   ├── body-baseline.png              # canonical body ref (optional)
│   │   ├── <slug>-NN.jpg                  # additional photo refs (optional)
│   │   └── _provenance.md                 # source URLs, capture notes
│   ├── locations/<slug>/
│   │   ├── _source.jpg                    # DAZ3D-scene-ref (env anchor)
│   │   ├── <slug>-NN.jpg                  # additional refs (optional)
│   │   └── _provenance.md
│   ├── props/<slug>/
│   │   ├── <slug>-NN.jpg
│   │   └── _provenance.md
│   └── style/<slug>/                      # optional, e.g., mood boards
│       └── ...
├── pages/
│   ├── panels/
│   │   ├── panel-01-<slug>/               # ONE FOLDER PER PANEL — NOT one file
│   │   │   ├── v1.png                     # first generation
│   │   │   ├── v1.notes.md                # (optional) why this was rejected
│   │   │   ├── v2.png                     # retry
│   │   │   ├── v2.notes.md
│   │   │   ├── v3.png                     # final retry
│   │   │   ├── v3.notes.md                # (optional) why accepted
│   │   │   └── _accepted.txt              # one line: "v3"
│   │   ├── panel-02-<slug>/
│   │   │   ├── v1.png
│   │   │   └── _accepted.txt              # "v1" — accepted on first try
│   │   └── ...
│   ├── page-01.png                        # lettered page from page-composer
│   ├── page-02.png
│   └── _index.md                          # page-composer's index
├── continuity-report.md                   # from continuity-check
├── posting/                               # populated by manual posting stage
│   ├── targets.json
│   ├── captions/page-01.md
│   └── posted.json
└── job_ids.md                             # chain log (Higgsfield runner OR Flow)
```

## Per-panel folder spec (the part this skill cares most about)

Each panel is a folder, not a single file. The folder name is `panel-<NN>-<slug>` where `NN` is the 2-digit panel number (01, 02, 03 — NOT panel_id like "p01-01" — keep the file system simple) and `<slug>` is a short kebab-case description that makes the folder browsable (e.g., `front-full-establishing`, `ecu-bicep-surge`, `splash-final-form`).

Inside each panel folder:

| File | Required | Purpose |
|---|---|---|
| `v1.png`, `v2.png`, … | yes (at least one) | Each attempt at generating this panel. Numbered in order of attempt. |
| `_accepted.txt` | yes (when an attempt is accepted) | Single-line file naming the accepted version: `v3`. Absence means "still in progress; nothing accepted yet". |
| `v<N>.notes.md` | optional | Why this version was rejected or accepted. Plain markdown. Common content: claude's auto-pick reasoning, observed issues (face drift, costume mismatch, anatomy issue), user's reason for retry. |
| `prompt-v<N>.txt` | optional | The exact prompt used for vN. Useful for debugging "why did v2 look like that?" without having to dig through the chat. |

### Why a folder per panel and not one file

- **Revision history is the work product.** Seeing that panel 3 took 3 attempts and panel 7 took 1 is itself valuable information for the user (and for retrospective lessons-learned). A flat file structure throws this history away.
- **The accepted version is explicit.** A symlink could also do this, but `_accepted.txt` is more portable across filesystems and shows up in git diffs as text.
- **Notes live next to their image.** A reviewer or future-self opening the folder sees vN.png + vN.notes.md side by side.
- **Composite generators can walk the structure deterministically.** `generate_composite.py --mode generation` reads `_accepted.txt` per folder to pick the right thumbnail, and reads the count of `v*.png` files to overlay "(3 attempts)".

### Migration from flat layout

If an existing project has flat panel files (`pages/panels/panel-01.png` instead of `pages/panels/panel-01/v1.png`), the skill treats each flat file as a single-revision panel: implied `v1` accepted on first try. Migration to folder structure is optional but recommended for any project that's still in active generation.

A one-shot migration script could be added later (move `panel-01.png` → `panel-01-<slug>/v1.png` and create `_accepted.txt`). Not built yet.

## How each skill writes into this layout

| Skill | Writes |
|---|---|
| `script-breakdown` | `shotlist.json`, `shotlist.md` |
| `reference-gathering` | `references/<bucket>/<slug>/` including `_provenance.md` and `_contact-sheet.md` |
| `comic-production` (Flow loop or Higgsfield runner) | `pages/panels/panel-NN-<slug>/v<N>.png`, `_accepted.txt`, optional `prompt-v<N>.txt` and `v<N>.notes.md`; updates `job_ids.md` |
| `continuity-check` | `continuity-report.md` |
| `page-composer` | `pages/page-NN.png`, `pages/_index.md`, optional `<project>.pdf` |
| (manual posting) | `posting/posted.json`, `posting/captions/page-NN.md` |
| `comic-status-board` | `STATUS.md`, `STATUS-*-board.png` (this skill — the only one that writes status artifacts) |

This division means: every other skill stays focused on producing its artifact. `comic-status-board` is the only skill that aggregates state into the user-facing view.

## A note on git tracking

For projects that are themselves git-tracked: the `STATUS.md` and `STATUS-*-board.png` files are auto-generated and may churn frequently. Two options:

1. **Track them in git** — useful if you want the status history committed alongside the project. The composites are PNGs (binary diffs), so commits get bigger; STATUS.md is text-friendly.
2. **Add to `.gitignore`** — useful if status is considered ephemeral working state. The project's source of truth (shotlist.json, panel folders) is still tracked; only the aggregated views are excluded.

Default recommendation: track `STATUS.md` (useful audit trail) but `.gitignore` the composite PNGs (regenerable, big, churn frequently). Up to the user per project.
