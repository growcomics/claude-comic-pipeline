# Comic Status Board

Produces visual progress reports for in-progress comics so the user can see what's happening without opening folders, file viewers, or external dashboards.

## Two principles (non-negotiable)

1. **Files live at the project root, never in buried subfolders.** Outputs are `<project>/STATUS.md`, `<project>/STATUS-references-board.png`, `<project>/STATUS-generation-board.png`, `<project>/STATUS-composition-board.png`. Anything generated by this skill is immediately discoverable in the project root.

2. **After writing files, Claude SURFACES them in the chat.** Every invocation of this skill must end with Claude reading the generated artifacts and displaying them inline in its response. STATUS.md → quote the relevant sections in the chat. Composite PNGs → use the `Read` tool on the saved PNG so the image appears inline in the chat. The user should see the status without needing to open anything.

## When to invoke

### Automatic (from `build-comic.md` orchestrator)
- After **references** stage completes → generate STATUS.md + the references-board composite
- After **generation** stage completes (all panels accepted) → generate STATUS.md + the generation-board composite
- After **composition** stage completes (all pages composed) → generate STATUS.md + the composition-board composite
- After **any single accepted panel** during generation → regenerate STATUS.md only (cheap; no composite)

### On demand (user)
- "Show me where we are" / "status" / "production board" → generate STATUS.md and surface it
- "Show me the refs" → generate the references-board composite if not present, else just surface the existing one
- "Show me the panels" → generate the generation-board composite (or surface existing one)
- "Show me the pages" → generate the composition-board composite

## Outputs

All written to `<project>/` (project root):

| File | Generated when | Contents |
|---|---|---|
| `STATUS.md` | After every references update OR every accepted panel | Markdown report with stages summary, inline thumbnail refs, panel-by-panel revision history, and links to the composites |
| `STATUS-references-board.png` | At end of references stage | Grid composite of all character/location/prop/style refs, labeled by bucket and slug |
| `STATUS-generation-board.png` | At end of generation stage (and on demand) | Panel sequence in story order, each panel shown at thumbnail size with an overlay strip showing revision count and accepted version |
| `STATUS-composition-board.png` | At end of composition stage | Full lettered pages laid out in reading order |

## How to invoke

### Generate STATUS.md
```bash
python scripts/generate_status.py <project_root>
```

Writes `<project_root>/STATUS.md`. Reads from `shotlist.json`, the `references/` folder tree, and the `pages/panels/` folder tree. Idempotent — safe to re-run.

### Generate a composite board
```bash
python scripts/generate_composite.py <project_root> --mode references
python scripts/generate_composite.py <project_root> --mode generation
python scripts/generate_composite.py <project_root> --mode composition
```

Writes `<project_root>/STATUS-<mode>-board.png`. Requires Pillow (`pip install Pillow`). Reads the same project state as `generate_status.py`.

## Folder convention (what the skill reads)

See `references/folder-convention.md` for the canonical layout. Key points:

- Each panel is its own FOLDER under `pages/panels/`, named `panel-NN-<slug>/`
- Inside each panel folder: `v1.png`, `v2.png`, etc. (revisions in order), plus `_accepted.txt` (one-line marker naming the accepted version like `v3`), plus optional `vN.notes.md` per revision explaining why a version was rejected/accepted
- References are in `references/characters/<slug>/`, `references/locations/<slug>/`, `references/props/<slug>/`, `references/style/<slug>/` per the `reference-gathering` skill's typed-bucket convention
- Pages from `page-composer` are at `pages/page-NN.png`

The skill enforces this convention. If a project uses a flat panel layout (`pages/panels/panel-01.png` instead of a folder), the skill treats it as a single-revision panel.

## The surfacing protocol (after every invocation)

After running the script(s), Claude MUST do the following in the same response:

1. **Read the generated STATUS.md** with the `Read` tool and quote the top-level stages summary + the most recent in-progress section (e.g., the in-progress panel's revision history) directly in the chat.
2. **For any composite PNG that was generated or already exists**: use the `Read` tool on the PNG file path so the image displays inline in the chat. Do NOT just mention that the file exists — actually show it.
3. **For ongoing work** (after each accepted panel, between stage boundaries): post a one-line status update referencing what changed since the last surface ("Panel 7 accepted on v2; 7/10 done. Generation board will refresh at end-of-stage.")
4. **Never assume the user has the project folder open in a file viewer.** Even if the user is local on the same machine, the chat is the canonical surface. If something was produced, it must be visible in the response.

## What the STATUS.md contains

```markdown
# <Project Name>
Last updated: <timestamp>

## Stages
- ✅ Script breakdown (10 panels, 4 cast, 1 hero location)
- ✅ References (1/1 character refs, 1/1 location refs, 0 props)
- 🔄 Generation (7/10 panels accepted)
- ⏳ Continuity check
- ⏳ Composition
- ⏳ Posting

## References (link to references-board.png if generated)

### Characters
- **chun-li** — ![face](references/characters/chunli/face-card.png) ![body](references/characters/chunli/body-baseline.png)

### Locations
- **chinese-alley-sunset** — ![source](references/locations/chinese-alley-sunset/_source.jpg)

### Props
(none)

## Generation Progress (link to generation-board.png if generated)

### Panel 01 — front-full establishing (size 1) ✅ accepted v3 (3 attempts)
- v1: face off-axis, rejected
- v2: costume tear position wrong, rejected
- **v3: accepted** — ![](pages/panels/panel-01-front-full/v3.png)

### Panel 02 — ecu-region bicep (size 2) ✅ accepted v1 (1 attempt)
- **v1: accepted on first try** — ![](pages/panels/panel-02-ecu-bicep/v1.png)

### Panel 03 — 3q-full mid-transformation (size 3) 🔄 in progress
- 4 variants pending review

[…and so on for all panels]

## Composition (link to composition-board.png if generated)
(pending)

## Posting
(pending)
```

The markdown is the authoritative human-readable view. The composites are the at-a-glance views for the three checkpoint moments.

## Future visual systems

Any future visual system (Antigravity multi-pane, custom local web dashboard, Paperclip integration) should read directly from the on-disk artifacts produced by this skill — `STATUS.md` and the three composite PNGs. The skill is the source of truth for project status visualization; renderers are interchangeable.

This means: when building a fancier dashboard later, that dashboard's only job is to *display* these files in a richer way. It shouldn't re-derive state from scratch — that's the skill's job, and the skill keeps STATUS.md current.

## Hard rules

- **Never bury outputs in subfolders.** All four output files live at the project root. No `_status/`, no `.status/`, no `status/`.
- **Always surface in chat after writing.** A successful invocation produces visible output in the chat response, not just files on disk.
- **Be idempotent.** Re-running on the same project state should produce the same outputs. No timestamps in filenames; the files are regenerated in place.
- **Don't fabricate status.** If the project has no `shotlist.json` yet, say so (`STATUS.md` shows "Script: pending"). Don't invent panels or refs that aren't on disk.
- **Don't move or modify project files.** Read-only access to everything except the four output files.

## Common asks

- **"Where are we?"** → run `generate_status.py`, surface STATUS.md
- **"Show me the refs"** → run `generate_composite.py --mode references`, surface the PNG
- **"How's panel 7 going?"** → run `generate_status.py` and quote just panel 7's section
- **"Final board"** → run all three composites + the final STATUS.md, surface everything

## Tools

- `python` with `Pillow` for composite rendering
- `Read` tool to surface generated files in chat
- `Bash` to invoke the scripts
- `Write` if the skill is generating STATUS.md directly without running a script (interactive mode)

If Pillow isn't installed, the composite scripts will fail with a clear error message; STATUS.md generation works without it.
